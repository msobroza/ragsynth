The ragsynth metric stack — an interactive guide

ragsynth generates test queries automatically, for evaluating RAG retrieval when labeled real queries are scarce. Its rule is validity over vibes: a generated test set is trusted only if it looks like real traffic (fidelity), covers what users ask (efficiency), and leads to the same decisions as real queries (validity). I · Why Why validating a generated test set matters

Your knowledge base changes every week, so a hand-labeled test set goes stale quickly. Generating queries from your own documents fixes freshness — but an LLM asked to "write a question about this chunk" produces polished paraphrases no real user would type.

The stakes are concrete: this test set decides which embedding model ships, whether a regression blocks a release, and whether quality is drifting. A wrong benchmark does not just waste effort — it points those decisions in the wrong direction, with confidence.

Concretely, a generated test set can mislead you in four specific ways. Each gets its own metric family — that is the structure of Part IV. Running example throughout: one real user question, "Is gradual water seepage covered by the contents policy?", and the chunk that answers it — extracted from an insurance policy PDF (the full page and dialogue follow in Part II):

the chunk that answers it · extracted from the policy PDF## Section 4 — Water damage Escape-of-water damage is covered when caused by a sudden burst of internal plumbing. Gradual seepage is excluded unless detected within 14 days.
failure mode 1 → fidelity

It doesn't sound like your users

Real queries are short, messy, and often only partly answerable; generated paraphrases are neither. If a simple classifier can separate the two clouds, the test set does not represent your users.

✓ "Is gradual water seepage covered by the contents policy?" — how a person asks.

✗ "Under which circumstances does Section 4's escape-of-water provision exclude gradually occurring seepage?" — how a chunk talks.

studied in: Chroma, Generative Benchmarking 2025 (polished, overly clean queries) · Penha et al., ECIR 2022 (real queries are messy)

It asks about the wrong things

Real demand is uneven — one toy-world topic is 40% of traffic. Uniform generation over-tests rare topics, under-tests frequent ones, and can leave topics with zero queries. The BCG rule: report coverage, not averages.

✓ "Is gradual water seepage covered by the contents policy?" — lands in the most-asked topic, in its real proportion.

✗ "How long should I simmer a tomato sauce?" — perfectly fluent, but no policyholder asks it: generated mass lands outside every topic real users care about.

studied in: Broestl et al. (BCG), arXiv 2510.00001 (semantic test coverage) · Coverage, Not Averages, arXiv 2604.20763

Efficiency — "the right topics, in the right proportions?" ↓
failure mode 3 → validity

It reaches different conclusions

A test set exists to compare systems and catch failures. If it ranks systems differently than real queries would — or shows no change when you damage the index on purpose — it creates false confidence.

✓ "Is gradual water seepage covered by the contents policy?" — the words don't match the chunk, so only the better retriever X finds Section 4 first. Real users see X win; the test should too.

✗ "Does Section 4 exclude gradual seepage not detected within 14 days?" — reuses the chunk's own words, so even the weak retriever Y nails it. Every system looks equal, the ranking flips, and you ship Y.

studied in: Rahmani et al., SIGIR 2024 (do synthetic collections rank systems like real ones?) · Sakai, SIGIR 2006 (does a test set detect real degradation?)

Validity — "would you make the same decision?" ↓
failure mode 4 → diversity

It repeats the same question

Many near-copies of one good query pass every test above. Simple diversity metrics catch the repetition.

✓ "Is seepage covered?" · "Which excess applies?" · "How fast must I report a claim?" — many questions.

✗ "Is gradual seepage covered?" rephrased four hundred times.

IV · Failure modes Four ways to be wrong — and the metrics that catch each

The four failure modes from Part I, now measured one by one. Each subsection opens with its question and closes with real v1 numbers; a summary collects all four into one picture before we present our method.

Failure modes · 1 — Fidelity Does the synthetic query cloud look like real traffic?

The idea: turn real and generated queries into points in one space — an embedding places similar meanings near each other — and try to tell the two clouds apart. Three tests of increasing strictness (KL profile, C2ST classifier, MMD kernel test), then the strictest: the same classifier inside each topic, where aggregate matching cannot hide anything.

Metric · KL divergence of query→corpus cosine profiles

Chroma's representativeness check. It compares one number per query — how close is it to its nearest chunk? Real users ask things the corpus only partly answers; generated paraphrases sit unnaturally close.

  1. For every query (real and synthetic), compute its top-1 cosine similarity to the whole chunk corpus: si = maxj qi·cj (embeddings L2-normalized).
  2. Histogram both sides on shared bins (50 in ragsynth), add a small ε to every bin so no bin is empty, normalize each side to sum to 1.
  3. Take the KL divergence of real (p) from synthetic (q). It is ≈ 0 when the two profiles match and grows as they separate. The measure is asymmetric on purpose: it strongly penalizes a generated set that has no queries where real traffic is common.
definition
KL(p∥q) = Σb pb lnpbqb
where b runs over the shared bins. Chroma's steered target on real data is roughly ≤ 0.16 — a reference band, not a hard gate.

Metric · C2ST & MMD — two-sample tests on the embedding cloud

One question, two tests: could anything tell the real and generated clouds apart? C2ST trains a classifier and reports its separation; MMD compares the clouds directly, with no training.

  1. C2ST (classifier two-sample test): label real = 0, synthetic = 1; fit a logistic regression with 5-fold cross-validation; report its ROC-AUC. AUC 0.5 means indistinguishable, 1.0 means trivially separable. A useful extra: the classifier's coefficients show which embedding dimensions reveal the difference.
  2. MMD (maximum mean discrepancy): measures the distance between the two clouds as wholes. It compares "how similar are points within each cloud" to "how similar are points across the clouds" — the result is zero exactly when the two distributions match. The kernel width is set automatically (median heuristic).
MMD² (unbiased) — kernel k(x,y) = e−γ‖x−y‖²
MMD2 = Σi≠jk(xi,xj)n(n−1) + Σi≠jk(yi,yj)m(m−1) − 2Σi,jk(xi,yj)nm
The intuition is the three averages, now shown live in the demo: ⟨k⟩ within real, ⟨k⟩ within synth, and ⟨k⟩ across. If the two distributions match, a point's neighbors across the clouds are as similar as its neighbors within — across ≈ within, and the three terms cancel to ≈ 0. Equivalently: the kernel turns each cloud into a smooth "density landscape" (its mean embedding); MMD is the distance between the two landscapes, and it grows exactly when across-similarity falls below within-similarity.

Metric · Within-cluster C2ST — the metric that separates A1 from A2

The metric that separates A1 from A2. Matching how many queries land in each topic says nothing about how they look inside it — so run the classifier separately inside every topic, where matched totals cannot help.

  1. Partition real traffic into C clusters once (frozen KMeans partition — every metric uses the same one).
  2. Assign each synthetic query to its cluster. For every cluster with ≥ 30 members on both sides, subsample to equal n and run a C2ST inside it.
  3. Report the mean AUC over scored clusters. ≈ 0.5 → synthetic queries are shaped like real ones within topics; ≈ 1.0 → they only match in aggregate.
why quotas can't fix this — the post-stratification residual
residual bias = Σc p̂c TV(p(·|c), q(·|c))
TV is total-variation distance — the largest difference the two distributions can assign to any event; p(·|c) and q(·|c) are the real and synthetic query distributions inside topic c. After re-weighting clusters perfectly, the bias that remains is exactly this within-cluster mismatch — the term A2's distribution-matched sampling attacks and this metric measures.

Failure modes · 2 — Efficiency & coverage Can this benchmark speak for production demand?

Fidelity asked whether generated queries look real; this block asks whether they are the right queries. Group real traffic into topics and measure each topic's demand share; then ask, in order: does every important topic appear at all (coverage — the BCG floors)? And does the mix match demand (weights & ESS)? The six symbols below are the entire language of this block:

Metric · Coverage — the BCG floor metrics

The simplest, strictest metric in the stack, from BCG's "Methodological Framework for Quantifying Semantic Test Coverage in RAG Systems" (Broestl et al., arXiv 2510.00001) and the companion rule "Coverage, Not Averages" (arXiv 2604.20763): before averaging anything, check which topics the test set covers at all. A topic with zero queries contributes nothing — no weighting can compensate.

  1. Embed test questions and corpus chunks into one shared semantic space and cluster it into topics. (BCG first filters outlier questions with a local-outlier-factor score, so irrelevant questions can't fake coverage.)
  2. Score how covered each topic is, then combine the topics with weights. Every coverage metric in this family — BCG's three and ragsynth's — is that same two-step skeleton; they differ only in which weight and which covered-ness.
  3. BCG weights a topic by its share of content (how many chunks it holds) and scores covered-ness softly (1 − average distance from each chunk to its nearest question). ragsynth weights a topic by its share of demand p̂c (how many real users ask about it) and scores covered-ness with a hard floor. Content-mass asks "is the corpus probed?"; demand asks "are the users represented?"
one skeleton, three instantiations — all scores in [0, 1], higher is better
Coverage = Σc ωc · cov(c)   — topic weight × topic covered-ness, summed over topics
varianttopic weight ωccovered-ness cov(c)
BCG content-weightedshare of chunks in topic c 1 − average distance from each chunk to its nearest question (soft, graded)
BCG multi-topicshare of chunks in topic c same, but a question only counts for a topic it sits near (within a threshold); a topic no question is near scores 0
ragsynth MSCkp̂c — share of real user queries in topic c 1[nc ≥ k] — all-or-nothing: 1 if the topic holds at least k synthetic queries, else 0
Read ragsynth's row aloud and it is one sentence: "what fraction of real user demand lands in topics the benchmark has at least k questions for?" At k = 1 this is demand-weighted coverage; the coverage gap is 1 − MSC₁ — the share of traffic the benchmark is silent about; the empty topics are listed by name (zero-query clusters), never averaged away. That is the dual-view rule "Coverage, Not Averages" enforces.
Where BCG's strategy sits on our ladder: the framework's remediation loop — find under-covered clusters, add questions there — is quota sampling in A1's structural sense, but stratified by corpus content instead of user demand: a content-stratified cousin of A1. It sits between A0 and A1: no region of the knowledge base stays empty, yet it remains demand-blind (a bulky appendix earns many questions; a hot topic with few chunks earns few), and it shares A1's blind spot — filling a cluster says nothing about the shape inside it (wC2ST), the gap A2's targets close. One parallel on the other side: BCG's LOF outlier filter is the post-hoc cousin of A2's reality check τr — both keep questions on the manifold of plausible ones, one by filtering after, one by rejecting targets before.
Roadmap — combining the two stratifications (spec v2.06): the two cluster sources answer different questions, so the plan is a hybrid: keep the demand map as the primary stratification and add a small, firewalled exploration slice — π′ = (1−ε)·demand-tilted map + ε·components routed to chunk clusters that fail a content floor (ε ≈ 0.10, targets from a hypothetical query per cold cluster, excluded from the demand headline). What it buys is early warning: a regression under a corpus region users have not queried yet is invisible to every demand-matched strategy by design — the planned drop_cold_cluster control makes that measurable, and only the hybrid detects it. Tested as a fifth strategy (A2X) in the harness; joint pooled query+chunk clustering is explicitly rejected (the modality gap splits clusters by type, not topic).
Worked example — why the average hides what coverage reveals Two topics: billing (60% of demand) and refunds (40%). The generated set has 100 billing queries scoring nDCG 0.95, and zero refund queries. The average over the generated queries is 0.95 — it looks excellent. The coverage is 0.60 — the set says nothing about 40% of what users ask. Both numbers are true; only one warns you. That is "coverage, not averages."

Metric · Importance weights, ESS & the post-stratified headline

Real traffic is uneven — toy topic 0 alone is 40% of demand. If the generated mix differs, each query is re-weighted by its topic's over- or under-representation. Re-weighting costs statistical power; these metrics measure the cost.

  1. Weights. Each synthetic query in cluster c gets wi = p̂c / qc — demand share over synthetic share. Balanced mix → all weights ≈ 1.
  2. ESS (effective sample size). Unequal weights waste data: 1000 queries with very unequal weights may carry the statistical power of only 550 (that is ESS/N = 0.55 — strategy A0's actual score).
  3. Coverage. Weights cannot fix a topic with zero queries — see the previous card.
  4. Post-stratified headline. Report the demand-weighted mean, and always alongside the per-cluster table (the dual-view rule) — a good average can hide one very poor topic.
the three identities
ESS = (Σiwi)2Σiwi2   ·   coverage = Σc p̂c 1[nc ≥ 1]   ·   M̂ = Σc p̂c mean(M | c)
symbols as in the notation box: p̂c = demand share of topic c, nc = synthetic count, wi = p̂c/qc. Variance inflation = n/ESS (Kong 1992): ESS/N = 0.5 means your benchmark is effectively half its size.

Metric · MAUVE (v2) — both failure directions in one curve

You now hold both failure directions: inventing mass where real traffic never goes (failure mode 1 — fidelity) and missing mass where real traffic lives (failure mode 2 — this section). MAUVE reads both in one curve — sweeping every trade-off between them and reporting the area: a precision–recall curve for distributions, with MAUVE as its AUC. It is the natural bridge between the two blocks.

  1. Embed both samples, quantize the shared space into bins (k-means in the real method; topics here), giving two histograms: real P, generated Q.
  2. Build a referee distribution between the two: Rλ = λP + (1−λ)Q, where λ ∈ (0, 1) is the mixing weight — at λ ≈ 1 the referee is almost the real P, at λ ≈ 0 almost the generated Q. Against this referee, compute two soft errors: KL(Q‖Rλ) — how much of Q is invented — and KL(P‖Rλ) — how much of P is missed. Comparing both to the mixture, not to each other, keeps the KLs finite even where one histogram has empty bins; each choice of λ weighs one error against the other.
  3. Plot the pairs (e−c·KL(Q‖R), e−c·KL(P‖R)) as λ sweeps 0→1: the divergence frontier. MAUVE = area under it. Identical distributions → the curve is the point (1,1) → area 1; the further apart, the more the frontier collapses toward the axes.
definition — the frontier as an integral
Rλ = λP + (1−λ)Q,   x(λ) = e−c·KL(Q∥Rλ),   y(λ) = e−c·KL(P∥Rλ)
MAUVE(P,Q) = ∫01 y(x) dx
the area under the parametric curve (x(λ), y(λ)) as the mixing weight λ sweeps 0→1; c > 0 is a scaling constant (c = 5 in the paper, smaller here for visibility). MAUVE ∈ (0, 1], and = 1 exactly when P = Q. The demo below computes this integral as a trapezoid sum over 41 values of λ.
reading the curve
Each λ contributes one point; sweeping λ from 0 to 1 traces the whole curve — λ plays the role the score threshold plays in a precision–recall curve. x-axis ≈ precision ("nothing invented"): shrinks when Q holds mass in bins where P has none. y-axis ≈ recall ("nothing missed"): shrinks when Q leaves P's bins empty. A hallucinating generator and a collapsing generator can have the same KL — but they bend this curve toward different axes.

Failure modes · 3 — Validity Does it rank retrieval systems the way real queries would?

The idea: fidelity and coverage are preparation; this is the goal. Score a set of retrieval systems on generated and on held-out real queries — same ranking (τ, τAP, RBO), plus detected injected failures (positive controls), means the test set can make decisions. If this block fails, nothing else matters.

Metric · Kendall τ, τAP, RBO — system-ranking agreement

Benchmarks exist to choose between systems. Rank a set of retrieval systems by nDCG — a 0-to-1 score, high when the right chunks rank near the top — on the generated set and on real anchor queries; if the orderings agree, the test set can substitute for real annotation.

  1. Kendall τ counts agreeing vs disagreeing pairs. Every pair of systems either keeps its order across the two rankings (concordant) or flips (discordant).
  2. τAP is τ with top-weighting: mistakes near the top of the ranking — where you actually pick a winner — cost more than mistakes at the bottom.
  3. RBO compares top-k prefixes with geometrically decaying weight (p = 0.9), robust when rankings only partially overlap.
  4. All three get bootstrap confidence intervals by resampling queries. The ORACLE strategy sets the ceiling: even two finite samples of real queries disagree a little.
definitions
τ = nconc − ndiscn(n−1)/2   ·   τAP = 2n−1 Σi=2n C(i)i−1 − 1   ·   RBO = (1−p) Σd=1 pd−1Ad
C(i) = how many systems ranked above position i are correctly above it; Ad = overlap fraction of the two top-d prefixes. Gates: τ ≥ 0.9 and τAP ≥ 0.8 ⇒ usable for model selection; below ⇒ regression-detection only.

      Metric · Positive controls — does the benchmark detect injected regressions?

      An injected regression is deliberate, known damage applied to the retrieval system — so you can check the test set notices damage you know is there. A test set that misses an injected failure is worse than none: it creates false confidence. ragsynth's battery injects three, each simulating a real production failure:

      injected regressionwhat it simulates in production
      drop_index(10%) — silently remove 10% of chunks from the index documents failing to sync; partial index corruption after a re-ingest
      noise(σ) — perturb every embedding with random noise a degraded, quantized, or silently swapped embedding model
      truncate_topk — cut retrieval depth an over-aggressive result limit or a timeout returning partial results
      1. Score every query before and after the injected degradation (paired design).
      2. Bootstrap the per-query deltas: resample queries with replacement, recompute the mean delta each time.
      3. One-sided p-value = fraction of resamples where the degraded system looks no worse. p < 0.05 ⇒ detected; p ≈ 1.0 ⇒ the benchmark is blind to this failure mode.
      paired bootstrap
      p = #{ resamples : mean(Δ*) ≤ 0 }B
      where Δi = score drop of query i under the injected degradation, B = 1000 resamples (Sakai's discriminative-power tradition).

      Failure modes · 4 — Diversity Is the set actually many different questions?

      The idea: a safety check. Nothing above notices one good question duplicated many times; simple repetition checks do.

      Metric · Distinct-n, dedup rate — and Vendi (v2)

      Every metric above can be passed by generating one good query many times. Diversity metrics detect that collapse.

      1. Distinct-1 / distinct-2: unique unigrams (bigrams) ÷ total. Low values ⇒ the generator is recycling phrasing.
      2. Semantic dedup rate: fraction of candidates removed for cosine ≥ 0.98 to an already-kept query — measured at the gate, where it doubles as a reject reason.
      3. Vendi score (v2): the "effective number of distinct samples" — build the pairwise similarity matrix, take its eigenvalue entropy. n identical queries → 1; n orthogonal queries → n. One number, interpretable as a count.
      Vendi score
      Vendi(K) = exp (−Σi λi ln λi)
      λ = eigenvalues of the normalized similarity matrix K/n. Since every diagonal entry of K is 1, they sum to exactly 1 — a probability distribution over the set's latent modes (its themes), where λi is the share of the set that mode i explains. Extremes: 12 duplicates → K is all ones → one eigenvalue = 1, the rest 0 → Vendi = e⁰ = 1. Twelve mutually dissimilar queries → K ≈ identity → twelve eigenvalues of 1/12 → Vendi = eln 12 = 12. Two tight themes of six → two eigenvalues near ½ → Vendi ≈ 2, which a plain average of pairwise similarities could never tell you. exp(entropy) is perplexity: the number of equally-weighted modes that would carry the same entropy.

      Coverage ↔ diversity — same family, orthogonal axes

      Related, and measurably so: coverage, ESS and Vendi are all effective counts — the same mathematical family (Hill numbers, qD = (Σi piq)1/(1−q)) at different orders, applied to different distributions. What separates them is what they count and relative to what.

      metriccounts whatrelative toHill order
      coverage / MSCkdemand clusters with support ≥ k production demand p̂q = 0 — richness: support only, proportions ignored
      ESSeffectively independent samples after re-weighting the importance-weight distributionq = 2 — inverse Simpson: 1/Σ w̃i2
      Vendieffectively distinct queries the set's own similarity spectrumq = 1 — exp-Shannon of kernel eigenvalues

      And why one can never substitute for the other — each is blind to the other's failure, by construction:

      1. Full coverage, zero diversity. One good query per topic, duplicated 50 times: coverage = 1.0, floors pass, ESS ideal — but only 8 questions exist. Only diversity objects (Vendi ≈ 8, dedup fires).
      2. High diversity, broken coverage. 400 different questions all in one topic: Vendi high, dedup clean — but coverage 0.40, gap 60% of demand. Only coverage objects.
      3. Coverage is between-topic and demand-relative; diversity is within-set and demand-blind. ESS sits between, checking proportions once support and distinctness hold.
      In ragsynth: that is why the report keeps all three families — the dual-view rule ("Coverage, Not Averages") governs what gets reported, the floors govern what counts as covered, and the diversity guard keeps both honest against repetition. The BCG framework's outlier-filtering step plays the mirror role on the other side: it removes questions that inflate diversity without covering anything.
      Hill 1973 (effective-number unification) · Friedman & Dieng, TMLR 2023 (Vendi as exp-entropy) · Kong 1992 (ESS) · Broestl et al., arXiv 2510.00001 (semantic test coverage)

      V · Our method A2 — distribution-matched sampling

      Paper order: related work first, proposition second. The instruments above can now judge any generator — the fidelity metrics detect shape mismatch, the efficiency metrics price it. Here is the method built to pass both at once.

      The idea, before the math

      how A2 corrects sampling bias — the polling analogy
      • A0 — street poll. Ask whoever walks by. Cheap, and biased.
      • A1 — quotas. Decide in advance: 50 answers from district A, 30 from district B. Right totals per district — but still whoever happened to be outside.
      • A2 — clone the census. First learn where real questions live (a map of real queries, weighted by demand). For each new query, pick a target spot on that map — throw it back if it lands where real traffic never goes — then write a query that lands there. One important subtlety: the map is soft. It is not hard districts with borders but a blend of overlapping bumps — a mixture model. A question sitting between two districts belongs a little to both, and targets are drawn from the whole blend — so the in-between questions real users actually ask get generated too, not only the district centers.
      The point: A2 fixes the bias before generating. The weights w = p̂/q come out ≈ 1, so there is nothing left to correct — while correcting afterwards by re-weighting costs about eKL extra samples. What A2 cannot fix — errors in its own map — is exactly what wC2ST measures.

      Inside A2 — the sampling algorithm, step by step

      The precise version of "clone the census": how one synthetic query gets its target before a single word is written.

      what is movMF — and why this distribution
      f(z; μ, κ) = Cd(κ) eκ μ·z   ·   p(z) = Σc π′c f(z; μc, κc)
      Embeddings are normalized, so queries live on the surface of a sphere — only their direction matters (cosine similarity is the dot product). A Gaussian is the wrong shape there: it puts probability off the sphere. The von Mises–Fisher (vMF) distribution is the sphere's Gaussian: a single bump centered on a direction μ (the topic's center), whose concentration κ plays the role of 1/σ² — large κ = a tight, specialized topic; small κ = a broad one. The dashed ellipses in the diagram are two such bumps. A mixture of vMF (movMF) is several bumps with weights π — the same idea as a Gaussian mixture, moved onto the sphere, fitted by EM (Banerjee et al., 2005; KMeans on cosine distance is its hard-assignment limit). Three reasons A2 uses it rather than plain clusters: (1) it is generative — unlike a KMeans partition, you can draw new targets z from it, which is the whole point of step 3; (2) every parameter is interpretable and steerable — π′ carries the demand tilt, μc the topic centers, κc the within-topic spread, which is precisely the "shape inside a topic" that quotas cannot match; (3) it is a small, frozen artifact — a few vectors and scalars, versioned per benchmark epoch, so the same demand map can be audited and reused.
      1. Map the demand. Embed all real queries and fit a mixture of "bumps" on the sphere (movMF): each bump has a center μc (a topic) and a spread κc. Weight each bump by its topic's demand share, softened by a mixing factor λ.
      2. Pick a topic. Draw a component c ~ Cat(π′) — frequent topics are drawn more often, in proportion to tilted demand.
      3. Pick a target. Draw a direction z ~ vMF(μc, κc): one precise spot inside the topic where a query should exist.
      4. Reality check. Keep z only if it is at least as close to some real query as real queries are to each other: maxj z·qj ≥ τr (the 5th percentile of real nearest-neighbor cosines). Otherwise throw it back and redraw — this is the rejection in the analogy.
      5. Collect evidence. Retrieve the chunks nearest to z — they become the seed and the gold-label candidates — plus nearby real queries as style examples.
      6. Generate and land. The LLM writes a candidate; embed it; if cos(emb(q), z) < τt, one revision asks it to move closer to the target. The survivor enters the verification gate like any other candidate.
      Why this corrects the bias: steps 1–4 make the targets follow the real query distribution — topic proportions and within-topic shape both — and step 6 forces each generated query onto its target. The output distribution q matches p̂ by construction, so the weights w ≈ 1 and nothing needs correcting afterwards.
      Banerjee et al., JMLR 2005 (movMF) · Wood 1994 (vMF sampling) · Morris et al., EMNLP 2023 (embedding-target lineage) · sampling/spec_sampler.py

      Why match at the source instead of re-weighting after? (the theory box)

      Three inequalities justify the whole design — they're cited in SPEC §16 as the theory anchors.

      1 · Pinsker: measured KL bounds headline bias for every system at once
      |Epm − Eqm| ≤ TV(p,q) ≤ KL(p∥q)/2
      for any bounded metric m ∈ [0,1]: if KL = 0.02, no system's headline can be off by more than √0.01 = 0.1 — and any decision between two systems with a margin > 2·TV is preserved. A low measured KL is a guarantee, not an impression.
      2 · Importance sampling cost is exponential in KL
      nneeded ≍ eKL(p∥q)
      Chatterjee & Diaconis 2018: correcting a mismatched distribution by re-weighting needs exponentially many samples. That's why A1/A2 match the demand at generation time rather than fixing it afterwards.
      3 · What quota matching leaves behind
      residual = Σc p̂c TV(p(·|c), q(·|c))
      post-stratification removes between-cluster mismatch entirely; the residual is purely within-cluster — measured by wC2ST, attacked by A2.

      Why it works — mechanism and payoff

      why A2 passes this test — the sampling algorithm, mapped to the metric
      The three-strategies demo in the fidelity section shows the picture; the algorithm above ↑ is the mechanism. wC2ST asks one thing of each topic: do the synthetic queries occupy the same region with the same spread as real ones? A1 cannot guarantee that — its generator sees a chunk and style examples, but nothing constrains where inside the topic its output lands, and LLMs drift to their own favorite phrasing: a tighter, shifted blob the classifier separates instantly (wC2ST 1.000). A2 constrains the landing point, step by step:
      algorithm stepwhat it buys inside a topic
      targets z ~ vMF(μc, κc) — with κc measured from real queries the synthetic cloud inherits the topic's true center and true dispersion — many targets reproduce the real spread, not the LLM's attractor
      reality check maxj z·qj ≥ τr no synthetic mass where real traffic never goes — no off-manifold tail for the classifier to key on
      target check cos(emb(q), z) ≥ τt + one revision each query actually lands on its own target, so the sampled spread survives generation instead of collapsing back to typical phrasing
      what remains: map-fit error and the landing tolerance the measured residual — wC2ST 0.680 instead of the 0.5 floor. This is the term the post-stratification residual formula (theory box above) names.
      The dispersion is load-bearing — v1 proved it by accident. An early toy A2 aimed every query at the topic's exemplar mean instead of sampling targets with the vMF spread: queries piled onto the center, the dedup check killed 60% of them, and fidelity collapsed. Restoring the measured dispersion (κ ≈ 400) is what produced the 0.680. Spread is not decoration; it is the mechanism.
      4 · One objective, two resolutions — why A2 gets fidelity and efficiency together
      Fidelity and efficiency are not two goals in tension: they are the same gap read at two zoom levels. Efficiency (weights, ESS, coverage) reads the coarse level — the topic totals. Fidelity (KL, C2ST, wC2ST, MAUVE) reads the full distribution, including the shape inside each topic. Matching the full distribution automatically matches its totals — so full fidelity implies full efficiency, never the reverse. A1 is the proof of the gap: ESS/N 0.90 with wC2ST 1.000. A2 aims at the single underlying objective — q ≈ p̂ at generation time — and the v1 table shows dominance, not trade-off: A2 matches A1's efficiency (0.91 vs 0.90) while strictly beating its fidelity. Two deliberate, bounded exceptions: the λ tilt gives rare topics slightly more than their demand share (a sliver of ESS traded for coverage floors), and the dedup guard caps how peaked q may become. Both are chosen knobs — not a fidelity-versus-efficiency dilemma.

      VI · Conclusion Key takeaways and next steps

      Key takeaways

      1. Trust is earned, not assumed. A generated test set must look like real traffic (fidelity), cover real demand (coverage & ESS), decide like real queries (validity), and not repeat itself (diversity).
      2. Matching proportions is not enough. A1 passed its quotas (ESS/N 0.90) yet was fully detectable inside every topic (wC2ST 1.000) and blind to an injected regression (p = 1.00). A2's distribution-matched sampling fixed both: wC2ST 0.680, τ 0.909, all controls detected.
      3. Coverage, not averages. Report the demand-weighted headline beside the per-topic table, and name the empty topics — the local-corpus run hid a 14% demand gap behind a healthy-looking average.
      4. Rankings, not scores. Absolute numbers inflate on synthetic collections (Rahmani et al.); trust the ordering — τ and τAP against a measured ORACLE ceiling.
      5. Always run positive controls. A test set that misses deliberate damage is worse than none: it creates false confidence.

      Next steps — the v2 roadmap

      1. Real-benchmark validation on FiQA-2018, NFCorpus and LegalBench-RAG — the run that decides which triggers below fire.
      2. Graded/pooled relevance judgments (UMBRELA) with judge calibration against ≥ 200 human labels — triggered if τAP stalls while τ passes.
      3. Closed-loop prompt optimization (MIPROv2, GEPA) steered by the gate's reject reasons — triggered when gate pass-rate and τ plateau.
      4. Richer meters and scale: Vendi and MAUVE (report-only), the soft demand map, parallel execution, and a ChromaDB backend.
      5. Beyond today's scope: the multi-turn stratum under MTRAG protocols, and visual grounding with CiteVQA-style masking ablation — see the annex below.

      Annex · Further work How the two excluded settings build their data

      CiteVQA — grounded QA pairs via masking ablation

      CiteVQA: Benchmarking Evidence Attribution for Trustworthy Document Intelligence (arXiv 2605.12882): 1,897 questions over 711 real PDFs — 7 domains, 2 languages, 40.6 pages per document on average. Each answer must cite the exact page region that supports it.

      1. Draft QA pairs over long real documents.
      2. Find each answer's crucial evidence automatically by masking ablation: hide a candidate region and re-ask. If the answer becomes impossible, that region is crucial — its element-level bounding box becomes the ground-truth citation.
      3. Expert reviewers validate the automated citations.
      4. Score with Strict Attributed Accuracy: a prediction counts only when the answer and the cited region are both correct.
      What ragsynth would borrow (roadmap R9): masking ablation as a generation-time filter — keep a generated query only if hiding its claimed evidence actually breaks the answer. It is the visual analogue of v1's round-trip gate check.
      arXiv:2605.12882

      MTRAG — a human-in-the-loop multi-turn benchmark

      MTRAG: A Multi-Turn Conversational Benchmark for Evaluating Retrieval-Augmented Generation Systems (Katsis et al., 2025): 110 conversations, 7.7 turns on average, 842 tasks, over four corpora — Wikipedia (CLAPnq), finance (FiQA), government sites, cloud documentation.

      1. Human annotators converse with a live RAG system (retriever + generator in the loop), so every next question reacts to a real answer.
      2. Per turn, annotators write the question with dimension labels, select the relevant passages, and repair the model's answer into a reference answer — 92% of turns needed repair.
      3. Difficulty is designed in: about 25% unanswerable questions, plus non-standalone follow-ups that reference earlier turns — the cases single-turn benchmarks never test.
      4. An automated companion (MTRAG-S) generates synthetic conversations over the same corpora — and falls measurably short: 5.9 vs 7.7 turns, 4.6 vs 16.9 unique passages per conversation, and it often produced partially answerable questions where truly unanswerable ones were required.
      What ragsynth would borrow (roadmap R8): the conversational stratum — dialogue generation with per-turn evidence, evaluated under MTRAG's protocols; the dialogue_context field activates then. MTRAG-S's shortfall is also the warning: multi-turn synthesis needs its own validity gates, not just single-turn ones.
      arXiv:2501.03468
